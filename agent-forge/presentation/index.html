CO-PILOT.

90 seconds to walk into a room and know what changed.
Chart-cited facts. No inference. No exceptions.

The Problem
90s
between rooms.

A physician has 90 seconds between patient rooms to recall who they're seeing, why, what's on file, and what actually matters today.

Right now they scan dense EHR notes, flip through labs, cross-reference med lists — under pressure, while the patient waits.

A trustworthy fact in 15 seconds beats a smart guess in 60.

The User

A primary care physician.
Multi-provider clinic. Polypharmacy panel.

PANEL

~2,000

Geriatric-leaning, multi-morbid. HTN, DM, CKD, AFib, hyperlipidemia.

DAY SHAPE

18–22 visits

15-minute slots, 9–12 and 1–5. Inbox window out of scope.

SURFACE

Desktop / WoW

Fixed workstation at each pod. Not optimizing for tablet.

WHY THIS USER

Highest-density follow-ups

Polypharmacy = legitimate drill-down questions. The right shape for an agent.

Surface A · Always-on

Pre-computed summary card

Four lines. Every claim cites the row it came from. Zero LLM call inside chart-open. This is the 90-second answer.

Used: the doorknob moment.
Surface B · Optional

Multi-turn chat drill-down

Same retrievals, same citations. Used when the doctor has 5–10 minutes. Ignored when they don't.

Used: pre-room and between-visit.
The Hard Boundary

Facts. From the chart. With citations.
Nothing else.

Surface chart-supported facts with row-level citations.
"Patient on metformin 1000mg BID" [src:prescriptions.4421]

Citation grammar — every claim, no exceptions.

Diagnoses or differentials
Treatment recommendations
Dose changes
Causal medical reasoning

Generation and reasoning are different products with different verification models. They are explicitly out of scope.

Audit Finding · 1 of 3
Security · A1
Authorization is role-coarse.
AclMain::aclCheckCore(section, value, user) // takes no resource ID. // "physician sees their own patients" — cannot // be enforced without new code. // REST `user/Patient.read` scopes gate capability, // not record identity.
Implication for the agent → Authorization must be enforced at the agent's own boundary. We cannot delegate to OpenEMR's ACL — it doesn't have the predicate we need.
Audit Finding · 2 of 3
Security · A2
No identity exists outside the request lifecycle.
$ rg "$_SESSION\['authUser" src/ library/ interface/ | wc -l 546 // 546 reads of session-bound user across the codebase. // No PSR-7 principal. No parameter-passed identity. // A backend invoked off the request thread has no // implicit user, no audit attribution, no facility scope.
Implication for the agent → Identity must be carried by a verifiable token — never by session, never by request body, never by an LLM tool argument.
Audit Finding · 3 of 3
Data Quality · D3–D8
The schema has no enforced referential integrity.
// Every "references" lives in column comments — not constraints. lists.type varchar -- polymorphic. no FK. no CHECK. -- keys problems / allergies / medications // Medications dual-stored: lists WHERE type='medication' AND prescriptions (separately) // Coded fields (RxNorm, ICD, SNOMED) — optional, frequently empty. // patient_data: NOT NULL DEFAULT '' on ~100 columns.
Implication for the agent → Every tool encodes the data quirks once, centrally, in clinical_filters.py. Soft-deletes, polymorphic types, dual storage, system rows — invisible to the LLM and the prompt author.
Architecture · 1 of 3

Two processes. One identity token.

PHP OpenEMR module · sessions, ACL, CSRF, Twig interface/modules/...
JWT HMAC-SHA256 · (user_id, pid, exp:+15m, jti) private network
PY Agent service · typed config, async SDKs, structured logs separate container

Why split. PHP speaks OpenEMR idioms. Python speaks agent idioms. One translation layer is the entire integration surface.

Why JWT. Wrong-patient prompt injection is closed cryptographically, not hopefully. Python tools read pid only from the verified token — never from request body or tool argument.

Cost paid: one extra hop, two deployables. Acceptable.

Architecture · 2 of 3

Ten typed SQL tools. No RAG.

The Use Cases

"What changed since last visit." "Any med interactions." "A1C trend." "What's new in last 30 days."

All structured-data questions. Truth lives in rows with dates, codes, IDs.

Why Not RAG

Cosine similarity on chunked prose is a fidelity-loss step over data that is already typed and queryable.

When use cases extend to free-text note search, RAG earns its keep. v1 doesn't go there.

Hardening

Read-only MySQL user. SELECT on clinical tables only. Parameterized queries.

Audit data quirks (D3/D4/D5/D8) baked once in clinical_filters.py.

Truth lives in rows. The agent reads rows.

Architecture · 3 of 3

Three-layer verification.
One citation grammar.

L1 Prompt design · grounded, cited output required model-side
L2 Deterministic post-processor · citation-existence + banned-phrase scan <50ms
L3 Independent verifier · different model family GPT-4.1-mini ⟂ Haiku
FALL Safe fallback · enumerated facts only, never silent strip "here are the rows"
Every claim: [src:table.id]
ASYMMETRIC TUNING

False-accept is the project-killer.

A recommendation pretending to be a fact harms patients. An over-eager verifier is UX cost. ≈2 orders of magnitude cost asymmetry → tune over-eager.

<1% false-accept · <10% false-reject
